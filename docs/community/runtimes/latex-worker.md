# Community LaTeX worker runtime

This page describes the optional Community worker for
`latex.compile.pdf/v1`. Interactive LaTeX preview continues to use BusyTeX in
the browser; the native worker runs only after an explicit **Build in
background** action and produces a durable artifact.

## Runtime decision

The Community worker uses a pinned native TeX Live distribution driven by
`latexmk`. It supports the existing project engine values `pdftex` and `xetex`
through native pdfLaTeX and XeLaTeX execution.

Tectonic is not the transparent implementation of either engine. It is a
XeTeX-derived system with its own compatibility and bundle semantics. A future
`latex.compile.tectonic.pdf/v1` operation may expose it as an explicit engine,
but substituting Tectonic behind a `pdftex` or `xetex` project setting would make
the processor contract misleading.

The native worker is valuable for a durable compatibility build. It does not
replace BusyTeX in the edit loop, and browser/native output is not promised to
match. The artifact records the runtime that actually produced it.

## Processor contract

The checked-in processor manifest is itself hashed to form the exact contract
advertised to Core. It records:

- operation and input-schema version;
- worker SDK/adapter revision and an implementation digest over workspace
  lockfiles, SDK and adapter source, Dockerfile, and `latexmkrc`;
- the pinned TeX Live base-image digest, release, and repository revision;
- `texlive.tlpdb`, filename-database, normalized fontconfig listing, and
  engine-format digests;
- TeX engine, `latexmk`, and bubblewrap versions;
- fixed command flags and environment policy;
- sandbox profile, fixed 2 GiB address-space limit, and artifact roles.

The year label alone is not a contract because a TeX Live repository continues
to update within a release. The current runtime is TeX Live 2026 revision 79639
with pdfTeX 1.40.29, XeTeX 0.999998, latexmk 4.88, and bubblewrap 0.11.2. The
image pins the base by digest and verifies both upstream-provided format files
and every declared runtime fingerprint during build. Production never runs
`tlmgr install` or `tlmgr update`.

## Image and process boundary

The runtime is an independently reusable OCI base layer:

```text
Community TeX Live runtime layer
  native engines and trusted helper programs
  read-only TEXMFDIST
  pre-generated TEXMFSYSVAR
  format files, font maps, ls-R, and fontconfig cache
  package and font manifests
              |
              v
Community LaTeX worker image
  worker agent and protocol SDK
  LaTeX processor adapter
  sandbox launcher
```

Changing the worker agent does not rebuild the TeX Live layer when its contract
is unchanged. Container registries and nodes reuse the content-addressed layer,
and the OS page cache naturally warms frequently read formats and packages. TeX
does not remain resident between jobs.

The final stage installs no packages from a rolling OS repository: all runtime
tools come from the pinned base snapshot. The Rust agent is PID 1, handles both
SIGINT and SIGTERM, drains its Core session, and waits for owned claim tasks to
close. This keeps process lifecycle and runtime provenance inside the same
versioned implementation instead of depending on an unpinned init package.

The worker image is optional and separate from the Community API image. A
deployment with no worker retains browser LaTeX behavior and does not carry the
native runtime size or idle execution capacity.

## Package profile

The canonical Community profile favors compatibility over minimum transfer
size and uses a pinned broad TeX Live image with the native engines and helper
programs needed by common projects. The pinned image digest and declared runtime
database hashes, not a human scheme name, are authoritative. An SBOM and
package/font license inventory are recommended release artifacts but are not
generated by the current repository checks.

An operator may build a smaller profile, but it produces a different processor
contract and must pass the same manifest and conformance checks. Missing
packages fail with a stable `package_unavailable` outcome. A user document
cannot trigger network installation, mutate a shared tree, or silently change
the runtime used by a queued job.

Package additions follow a promotion workflow:

1. update the frozen build input or package selection;
2. rebuild formats, maps, filename databases, and font caches;
3. generate and verify the new runtime manifest;
4. run the compatibility and security corpus;
5. publish a new immutable runtime and processor contract;
6. roll worker capacity without changing Core or existing artifacts.

## Four cache scopes

The word cache covers four different states with different safety rules.

| State | Reuse scope | Policy |
| --- | --- | --- |
| TeX Live package/runtime tree | all jobs using one contract | immutable, read-only OCI layer |
| formats, maps, filename and font caches | all jobs using one contract | generated during image build, read-only |
| project auxiliary and `latexmk` state | one attempt | writable per-job directory, deleted at end |
| final PDF and deterministic artifacts | exact cache identity | immutable blob reused through a new authorized job |

The package tree is runtime content rather than a mutable job cache. Kpathsea's
pre-generated `ls-R` avoids recursive filesystem scans. System formats and maps
live in the immutable system variable tree.

Each job receives private writable paths for `HOME`, `TEXMFVAR`,
`TEXMFCONFIG`, `TEXMFCACHE`, `TEXMFOUTPUT`, build output, and temporary files.
`TEXMFHOME` points to an empty controlled tree. Any lazily generated data is
therefore destroyed with the job rather than crossing a user or project
boundary.

The initial worker does not reuse `.aux`, `.toc`, `.out`, `.bbl`, `.bcf`,
`.run.xml`, `.fdb_latexmk`, `.fls`, or SyncTeX state across jobs. Such reuse
would require project affinity, complete invalidation, confidentiality controls,
claim fencing, quota, and cleanup. Background work does not justify that
complexity before measurement. An exact repeated input is faster and safer to
serve from the final artifact cache.

The artifact-cache identity is reuse scope, operation version, immutable input
digest, processor contract, and normalized options digest. The initial LaTeX
operation scopes reuse to the source project. A hit creates a new job and
authorized artifact reference; it never grants access to the job that first
created the shared blob.

## Separation from the browser TeX Live proxy

BusyTeX requests individual TeX Live files through an authenticated backend
proxy and a versioned on-demand cache. That cache is not a native `TEXMFDIST`:
it does not promise the full directory hierarchy, filename database, native
tools, system formats, or map state required by Kpathsea and `latexmk`.

The browser and native worker have independent runtime contracts. BusyTeX
fetches its configured TeX Live 2026 assets, while the native image pins an
immutable TeX Live 2026 snapshot by manifest digest and verifies its package
database, file database, font listing, engines, and formats while building.
They still do not promise identical package snapshots or output and do not
mount the same writable cache. Converting an incidental browser request cache
into the native runtime would couple correctness to access history and mix two
ownership and retention models.

## Per-job filesystem

Bubblewrap receives controlled mounts and a private job tree:

```text
/usr                    read-only runtime and TeX Live tree
/etc/texmf              read-only TeX configuration when present
/etc/fonts              read-only font configuration when present
/runtime/latexmkrc      read-only contract-owned recipe
/job/work               private writable copy of verified project input
/job/build              writable auxiliary state
/job/texmf-var          writable personal runtime state
/job/texmf-config       writable personal configuration state
/job/texmf-cache        writable personal cache state
/job/home               empty writable home
/job/tmp                private temporary files
/proc                   empty directory; procfs is not exposed to the job
```

The SDK verifies the immutable input and then creates a private copy for
compatibility with packages that write near
their sources. All such changes disappear with the attempt and cannot alter the
recorded input digest. Project files may intentionally shadow package names
within their own work root, as normal TeX projects do, but they cannot write
into or replace the runtime tree. Paths, symlinks, hard links, devices,
duplicate archive entries, expanded size, and file count are validated before
execution.

## Build invocation

The adapter selects the command from the frozen project engine; user input is
never concatenated into a shell command.

- `pdftex` selects a fixed `latexmk` pdfLaTeX recipe.
- `xetex` selects a fixed `latexmk` XeLaTeX recipe.
- the entry file is a validated project-relative path prefixed with `./`, so
  even a basename beginning with `-` cannot be parsed as a `latexmk` option;
- automatic system, user, and project `latexmkrc` discovery is disabled; only
  the immutable worker-owned recipe is loaded;
- interaction is noninteractive and errors produce a nonzero status;
- shell escape is disabled at engine and TeX Live configuration levels;
- locale, timezone, source date, paper defaults, search paths, and output paths
  are explicit;
- network proxy variables and inherited search/config variables are removed;
- process creation, wall time, memory, CPU, and output are sandbox-bounded.

`latexmk` may run only the trusted bibliography, index, and conversion programs
included by the processor profile and necessary for the selected recipe. A
document cannot add an executable to `PATH` or enable arbitrary `\write18`.
Supporting a workflow such as minted requires a separate reviewed operation or
processor profile; it is not enabled by loosening the default sandbox.

## Sandboxing

Native TeX and its helper programs process untrusted, complex formats. Command
flags are defense in depth, not containment. The production profile runs each
attempt in an OS isolation boundary with:

- non-root user and read-only root;
- no worker credential, database URL, object-store secret, or service-account
  token;
- no network;
- bounded CPU, memory, process count, open files, temporary storage, and time;
- a syscall and capability profile appropriate to the runtime;
- private process and writable filesystem state;
- termination on cancellation or before the last confirmed lease expires.

The worker agent downloads and verifies input before entering the sandbox and
uploads results after it exits. Bubblewrap provides private PID, mount, network,
IPC, and UTS namespaces; `prlimit` bounds address space, CPU seconds, processes,
open files, and output size; Core owns wall timeout and lease cancellation. The
job does not need procfs, so its private PID namespace is represented by an
empty `/proc` instead of requiring the outer container to unmask host system
paths. The outer deployment bounds `/work`, which is also the SDK temporary
directory.

The Compose profile uses `seccomp=unconfined` only to make rootless namespace
setup portable for local validation. On AppArmor hosts, the source-controlled
`toss-latex-worker` profile admits only the dedicated `toss-bwrap` setup path
and stacks a capability-stripping profile onto its child. Production must
provision and review both its MAC and syscall policies instead of copying the
local seccomp setting. The `process` executor is for trusted development only
and advertises a deliberately different processor contract, so it cannot
satisfy the production allowlist accidentally.

## Output and diagnostics

The primary result must begin with a valid PDF signature, remain within the
declared size limit, and pass Core's bounded header/trailer checks before
publication. The only optional role in version 1 is a sanitized text compile
log; SyncTeX is not part of this operation.

Diagnostics are truncated on character boundaries and normalize the source,
build, runtime, home, and host paths to stable placeholders. They never include
environment variables, command lines containing secrets, transfer URLs, object
keys, or arbitrary binary output. Full raw logs remain inside the disposable
sandbox unless an explicit protected diagnostic policy is designed later.

Compile errors, missing packages, invalid input, timeout, output limit, worker
interruption, transfer failure, and internal contract failure have distinct
machine classes. The worker reports evidence; Core decides retry policy.

## Determinism and artifact reuse

The adapter fixes all controllable environment and command inputs, including
`SOURCE_DATE_EPOCH`, timezone, locale, and normalized output paths.
`SOURCE_DATE_EPOCH` comes from the stable project-bundle manifest covered by the
input digest, never from claim or worker start time. The runtime manifest
captures the pinned runtime and font-index identity.

The current operation treats an exact input, options, and processor-contract
tuple as reusable and returns the first published artifact through a new
authorized job. It does not claim that a fresh second TeX invocation would be
byte-identical. If that reuse contract becomes invalid for a runtime, release a
new operation/processor policy rather than silently changing the existing
contract.

## Limits and degradation

Deployment policy owns slots, queue depth, input bytes, PDF and diagnostic
bytes, wall time, attempt count, and retention. The SDK additionally enforces a
fixed 4,096-file and 256 MiB expanded-bundle ceiling. Attempt rows record the
wall/output/diagnostic limits supplied to a claim.

A full or offline worker leaves configured builds queued or waiting under
Document Processing policy; an unconfigured operation is omitted. Browser
BusyTeX preview and local export continue independently. The UI never silently
turns a failed durable build into a successful browser build.

## Verification

CI currently checks Rust formatting/lints/tests, the exact Core bundle shape,
path and manifest rejection, missing-package classification, the source and
Dockerfile implementation digest, generated contracts, and fake-worker
protocol behavior against a disposable backend. Docker build pins the TeX Live
base and verifies the repository revision, package and filename databases,
fontconfig listing, tool versions, and both engine formats.

Release validation must additionally run the built image through Core and
compile both pdfLaTeX and XeLaTeX projects inside bubblewrap, download the
published PDFs, and confirm cache reuse. The repository's
`scripts/processing-latex-worker-smoke.mjs` automates that release check with
multi-pass documents containing Unicode/fonts, references, tables, mathematics,
and vector graphics. BibTeX or Biber, indexes, external images, shell-escape
attempts, archive bombs, timeout, cancellation, and deterministic-fixture
repetition remain the next compatibility and adversarial corpus.

### Browser/native performance baseline

A 2026-07-14 local run of
`scripts/processing-latex-benchmark.mjs` compared the actual product paths
on an 8-logical-CPU Cascade Lake host with 15.6 GiB memory and Chrome 149. The
browser used BusyTeX 1.2.3 with TeX Live 2026; the worker used the former native
TeX Live 2025 snapshot, latexmk 4.87, and bubblewrap. The benchmark predates the
capacity-offer protocol and the worker's TeX Live 2026 upgrade, so its exact
numbers remain historical rather than a benchmark of the current image.
Both compiled the same synthetic public corpus: about 21 KiB, 36 generated
workload blocks, a table of contents, cross references, long tables,
mathematics, TikZ, hyperlinks, and fonts.

| Engine and path | Compiler or claim-to-delivery median (range) | User or HTTP-to-success median (range) |
| --- | ---: | ---: |
| pdfTeX, new browser context | 8.48 s (8.43–27.65) | 9.54 s (9.47–28.80) |
| pdfTeX, persistent BusyTeX worker | 4.75 s (4.52–4.81) | 4.96 s (4.76–5.05) |
| pdfTeX, native clean job | 3.44 s (3.33–3.59) | 4.03 s (4.00–18.06) |
| pdfTeX, exact artifact cache | — | 1.03 s |
| XeTeX, new browser context | 13.73 s (13.29–19.06) | 14.83 s (14.31–20.14) |
| XeTeX, persistent BusyTeX worker | 9.38 s (9.21–9.53) | 9.66 s (9.47–9.73) |
| XeTeX, native clean job | 5.30 s (5.16–6.36) | 6.86 s (6.04–18.05) |
| XeTeX, exact artifact cache | — | 1.02 s |

New-context rows have three samples; persistent and native-clean rows have five.
The first browser run began with an empty Core TeX Live proxy cache. Each
deliberately cache-empty browser context transferred about 81 MB across 107–110
runtime responses; persistent-worker edits transferred no additional runtime
bytes. Native jobs used a warm immutable runtime/page cache but a new private
job tree with no auxiliary-file reuse, which is the intended steady-state
worker model.

Native execution itself was about 27% faster than warm BusyTeX for pdfTeX and
44% faster for XeTeX in this fixture. That does not make the native worker a
better interactive preview path: it consumes shared server capacity and adds
snapshot, queue, transfer, sandbox, publication, and availability behavior.
Its value remains compatibility, durability, provenance, and reusable
artifacts.

The wide native end-to-end ranges exposed a scheduling defect rather than TeX
variance. The old protocol persisted heartbeat-reported spare capacity, so a
completed claim could wait for the next 15-second session heartbeat before the
worker became eligible again. Three samples therefore waited about 10–14
seconds before claim start even though claim-to-delivery time stayed stable.
Those end-to-end ranges describe the pre-fix scheduler and must not be used as
a current service objective.

The scheduler now has one clean capacity model. Session registration declares
the processor's slot ceiling, Core transactionally enforces that ceiling
against unexpired running claims, and the agent offers capacity only after
reserving local semaphore permits. Session heartbeat carries health, not spare
capacity. A disposable-PostgreSQL protocol regression held one claim active,
sent a session heartbeat, completed it, and reacquired in 25 ms; two concurrent
acquisitions against the one-slot session produced exactly one claim. A fresh
image-level smoke run of the final TeX Live 2026 image then compiled both real
engines and hit the exact artifact cache through the new agent protocol. Its
first delivery-to-next-claim gap was 1.02 seconds including the client-side
success observation and next job submission. SIGTERM produced exit code zero
and left the worker session in `draining`. This removes heartbeat cadence from
scheduling latency without changing the decision to keep the worker off the
editor's latency-critical path.
